# 银行内部 AI 企业智能助手面试手册

这份手册的目标不是把代码逐行背下来，而是让你能在面试中清楚讲出：

1. 项目解决什么业务问题。
2. 为什么这样设计。
3. 核心链路怎么跑。
4. RAG、文档解析、Function Calling、安全合规、性能高可用这些点如何落地。
5. 面试官追问时如何回答得有深度、但不夸大。

## 1. 30 秒项目介绍

可以这样说：

> 这个项目是一个面向银行内部员工的 AI 企业智能助手，核心目标是让员工通过自然语言完成两类工作：第一类是制度问答，比如查询制度、操作手册、FAQ；第二类是办公办理，比如请假、薪资查询、审批处理、外包入场申请。系统基于 Spring Boot 3、MyBatis、MySQL、Redis、RabbitMQ、Elasticsearch、阿里云 OSS、Spring AI、LangChain4j 和通义千问实现。整体采用 RAG + Function Calling 的架构，知识库文件上传后会异步解析、切片、向量化并写入 ES；用户提问时通过意图识别路由到 RAG 检索、业务查询、业务办理或闲聊。写操作全部走二次确认和后端安全沙箱，所有关键行为都有审计日志和风险控制。

## 2. 2 分钟项目总述

可以这样展开：

> 这个系统从业务上分为知识库问答和业务办理两条主线。
>
> 知识库这边，管理员上传 PDF、Word 制度文件，系统先校验文件类型和大小，再计算 SHA-256 做去重，然后上传到 OSS，元数据落 MySQL，并发送 RabbitMQ 消息触发异步解析。解析服务从 OSS 下载文件，用 PDFBox 和 POI 提取内容，做文本清洗、标题识别、表格处理和智能切片。切片会保存到 MySQL，同时调用通义 text-embedding-v2 生成 1536 维向量，再批量写入 Elasticsearch。检索时采用 BM25 + 向量 KNN 的混合检索，之后用 RRF 做融合，再做查询改写、本地重排、权限过滤和引用溯源。
>
> 对话这边，用户消息进入对话入口后，系统会加载 Redis 中的多轮上下文，先判断是否处于补槽或待确认状态。如果不是，就调用通义千问做意图识别，分为制度问答、业务查询、业务办理、闲聊和信息不完整。制度问答走 RAG，业务查询走读工具，业务办理走写工具并生成待确认操作。写操作不会直接执行，必须用户确认后，后端再次校验权限、指令归属和业务规则，再调用真实业务系统适配层。
>
> 安全上做了三层防线：意图级权限预检、AI 生成指令加用户二次确认、后端落库前再次校验和全链路审计。性能上做了线程池隔离、检索并发舱壁、Embedding 短 TTL 缓存、检索超时降级，以及 Actuator 和 Prometheus 指标。

## 4. 总体架构怎么画

面试白板可以这样画：

```text
前端/员工
   |
   v
对话入口 DialogController
   |
   +-- Redis 会话记忆
   +-- 意图识别 Qwen
   |
   +-- POLICY_QA ------> RAG 混合检索 ------> ES + Embedding
   |
   +-- BUSINESS_QUERY -> Function Calling 读工具 -> 业务系统 Client
   |
   +-- BUSINESS_EXECUTE -> Function Calling 写工具
                         -> 待确认操作 Redis/MySQL
                         -> 用户确认
                         -> 安全沙箱
                         -> 真实业务系统 Client
   |
   +-- 审计日志 / 安全日志 / 指标
```

文档知识库生产链路：

```text
上传文件
  -> 文件校验
  -> SHA-256 去重
  -> 上传 OSS
  -> 文档元数据入库
  -> RabbitMQ 解析消息
  -> 下载 OSS 文件
  -> PDF/Word 解析
  -> 清洗和切片
  -> 切片入 MySQL
  -> Embedding
  -> ES Bulk 写入
  -> 解析完成/失败
```

## 5. 核心模块讲法

### 5.1 文档上传模块

面试官可能问：文件上传怎么保证一致性？

回答：

> 上传接口先做文件非空、类型和大小校验，只允许 PDF、DOC、DOCX，大小限制 50MB。然后计算 SHA-256 哈希做去重，如果数据库已经存在相同 hash，就直接返回已有文档 ID。真正上传时先生成 UUID 文档 ID 和 OSS 路径，上传 OSS 后再在事务中保存元数据。如果数据库保存失败但 OSS 已上传成功，会清理 OSS 对象，避免出现数据库没有记录但 OSS 有脏文件的情况。上传成功后通过事务后回调发送 RabbitMQ 解析消息，消息发送失败只记录日志，不影响上传结果，后续可以补偿重发。

重点词：

- SHA-256 去重
- OSS 对象路径规范
- `@Transactional`
- OSS 成功但 DB 失败的补偿清理
- 上传完成后异步解析

### 5.2 RabbitMQ 异步解析模块

面试官可能问：为什么要用 MQ？

回答：

> 文档解析和向量化比较耗时，尤其 PDF 页数多、表格复杂或者需要 OCR 时，如果同步做会拖慢上传接口，也容易超时。所以我把上传和解析解耦，上传只负责文件和元数据落地，解析通过 RabbitMQ 异步触发。消费者使用手动 ACK，先检查文档状态做幂等，如果已经解析中或解析完成就直接 ACK。解析失败会根据重试次数决定重新投递或进入死信队列，超过最大重试次数后更新文档状态为失败，并记录告警日志。

追问：Nack 重新入队有什么风险？

回答：

> 如果直接 nack requeue，可能短时间内反复消费同一条失败消息，造成空转。所以消息体里带当前重试次数，业务层重新发送带新次数的消息；达到上限后 nack 不重新入队，让消息进入 DLQ。生产上还可以用延迟队列或 TTL + DLX 做退避重试。

### 5.3 PDF/Word 解析、清洗、切片

面试官可能问：切片怎么做，为什么不直接按固定长度切？

回答：

> 固定长度切片容易把制度条款切断，后续检索虽然能召回，但回答引用会不完整。所以切片策略优先按文档结构切。PDF 解析时会逐页提取文本、标题、表格和页码，识别目录页并过滤。清洗阶段会处理页眉页脚、页码噪声、多余空白、异常换行和全角半角。切片阶段优先用标题作为边界，一个标题及其下属内容作为切片单元，并保留完整章节路径，比如“员工考勤管理办法 -> 第三章 审批流程 -> 第五条”。如果内容超过目标长度，就在段落边界二次切分，并保留一定重叠，防止上下文断裂。表格会单独转成 Markdown 切片。

追问：标题怎么识别？

回答：

> 目前结合了编号模式和版式特征，例如“第一章”“第一条”“1.1”这类制度常见编号，也考虑字体大小、加粗等信息。Word 解析通过 POI 获取段落和表格，PDF 通过 PDFBox 做位置感知文本提取。这个模块后续最依赖真实制度样本继续调优。

### 5.4 向量化与 Elasticsearch

面试官可能问：ES 索引怎么设计？

回答：

> 知识切片索引里既有过滤字段，也有检索字段。文档 ID、文档名称、文档类型、版本号、部门、密级、状态等是 keyword，用于权限过滤和版本过滤；切片正文是 text，配 IK 分词；标题路径也是 text，用于标题加权；向量字段是 dense_vector，维度 1536，和通义 text-embedding-v2 对齐，使用余弦相似度。写入时使用 BulkRequest 批量入库，部分失败会记录日志，避免整批失败影响所有切片。

追问：为什么数据库和 ES 都存切片？

回答：

> MySQL 是权威存储，用于审计、管理、重建索引和问题排查；ES 是检索加速层。ES 可以删除重建，MySQL 切片仍然保留。解析流程里也是先保存切片，再写 ES，ES 写入成功后才把文档状态更新为解析完成。

### 5.5 RAG 混合检索

面试官可能问：你们的 RAG 怎么实现？

回答：

> 在线检索同时走两路召回。一路是 BM25，用用户问题在切片正文和标题路径上做全文检索；另一路是向量检索，先对用户问题生成 embedding，再用 ES dense_vector 做 KNN。两路各取 Top 50，然后用 RRF 倒数排名融合，公式是 `score = sum(1 / (60 + rank))`。融合后再做标题路径命中加权、本地重排、相似度阈值过滤，并生成引用来源和高亮片段。

追问：为什么要混合检索？

回答：

> 制度问答里既有语义相似问题，也有很多精确词，比如制度名称、条款编号、部门、流程节点。纯向量检索对语义泛化好，但对专有名词和编号不一定稳定；BM25 对关键词精确匹配强，但同义表达弱。混合检索能互补，RRF 不依赖两路分数尺度一致，比较适合融合 BM25 和向量检索。

追问：RRF 为什么不用直接加权分数？

回答：

> BM25 分数和向量相似度不是同一个量纲，直接加权需要大量调参。RRF 基于排名，不依赖原始分数尺度，更稳，也更容易解释。只要一个切片在任一路召回里排名靠前，融合后就会得到合理提升。

追问：如何降低幻觉？

回答：

> 回答 Prompt 要求只基于检索片段作答，不足时明确说知识库中未找到足够依据。响应里会带引用来源，例如文档名、章节路径、页码。低于阈值时直接返回低置信提示，不让模型硬答。后续还可以增加强引用校验和答案后处理。

### 5.6 权限过滤

面试官可能问：知识库权限怎么控制？

回答：

> 权限过滤是在 ES 查询层实现的，而不是查出来后再过滤。系统会根据当前用户角色和部门构建 BoolQuery filter，自动限制只查已发布、最新版本、用户可访问密级和部门范围内的文档。密级规则是公开所有人可见，内部登录用户可见，机密和秘密需要对应角色；部门规则是非管理员只能看自己部门或无部门限制的文档。这样可以减少敏感切片出现在应用内存中的概率。

### 5.7 意图识别与路由

面试官可能问：用户问题如何路由？

回答：

> 对话入口先加载 Redis 会话状态。如果当前处于等待补槽，就不重新识别意图，直接补槽；如果处于等待确认，就直接走确认/取消逻辑。正常状态下调用 Qwen 做意图分类，输出结构化 JSON，包括 intent、confidence、slots、missingSlots、clarifyQuestion 和 routeTarget。意图分为 POLICY_QA、BUSINESS_QUERY、BUSINESS_EXECUTE、CHITCHAT、AMBIGUOUS。识别失败或超时时有规则兜底，默认走 POLICY_QA，这是比较保守的策略，因为制度问答是只读链路。

### 5.8 Function Calling 工具层

面试官可能问：业务工具如何保证安全？

回答：

> 工具分 READ 和 WRITE。READ 工具权限校验通过后可以直接查业务数据，比如假期余额、薪资明细、审批进度。WRITE 工具不会直接执行业务写入，只会生成待确认操作，写入 Redis 和 MySQL，前端展示确认卡片。用户确认后，后端再次从 Redis 读取待确认指令，校验当前用户是创建者、指令未过期、当前仍有权限，并通过业务规则校验后才调用真实业务系统 Client。

追问：为什么前端不能直接调用业务接口？

回答：

> 为了防绕过。前端只能调用确认接口，真实业务 API 不直接暴露给前端。确认接口内部重新校验 Redis 指令、用户归属和权限，避免用户篡改 pendingId 或绕过 AI 流程直接提交写操作。

### 5.9 SSE 流式对话和前端协议

面试官可能问：流式返回怎么设计？

回答：

> 后端使用 SseEmitter 返回 `text/event-stream`。每个事件都用统一 envelope，包含 eventId、type、conversationId、messageId、sequence、payload、timestamp。事件类型包括 conversation_start、status、message、retrieval_result、tool_start、tool_result、confirm_required、slot_required、done、error。这样前端可以按 sequence 追加 token，也能根据 messageId 做幂等和链路追踪。

### 5.10 安全合规

面试官可能问：银行场景下你做了哪些安全设计？

回答：

> 主要有三层安全沙箱。第一层是意图识别后的权限预检，无权限请求不会进入工具链路。第二层是写操作二次确认，AI 不能直接改业务数据。第三层是用户确认后后端再次校验权限、指令归属和业务规则，再调用真实业务系统。同时所有意图、工具调用、确认、取消、检索和安全拒绝都会写审计日志。审计日志会对薪资、身份证、手机号、银行卡等敏感信息脱敏，记录 traceId、风险等级、客户端 IP 和留存时间。还有定时对账任务，检查 AI 确认执行表和业务订单表是否一致。

### 5.11 性能与高可用

面试官可能问：性能上做了什么？

回答：

> 在线检索用了独立线程池并发执行 BM25 和向量召回，避免占用 Web 容器线程。对 RAG 检索加了并发舱壁，超过最大并发会快速降级，避免把 ES 和 Embedding 服务打爆。BM25 和向量检索都有超时降级，单路失败不会拖垮整体。Embedding 对用户查询做了本地短 TTL 缓存，减少重复向量化调用。系统还接入 Actuator、Micrometer 和 Prometheus，暴露检索耗时、命中数、请求结果分类、舱壁剩余许可和缓存大小。

## 6. 面试高频问题与参考回答

### Q1：这个项目最大的亮点是什么？

回答：

> 我觉得最大亮点是把 RAG 问答和业务办理放在一个统一对话入口里，但又对读写操作做了严格隔离。制度问答走知识库检索和引用溯源，业务查询走只读工具，业务办理走写工具但必须二次确认和后端安全沙箱。这样既有 AI 助手的自然语言体验，又能满足银行内部对权限、安全、审计和可追溯的要求。

### Q2：为什么选择 MyBatis 而不是 JPA？

回答：

> 这个项目后续会涉及比较多审计表、查询筛选、状态流转和与既有数据库规范对齐的 SQL。MyBatis 对 SQL 可控性更强，也更符合很多银行内部系统的开发习惯。表结构由 Flyway 管理，Mapper 用注解方式实现，简单查询足够轻量；复杂查询也方便后续迁移到 XML。

### Q3：文档解析失败怎么办？

回答：

> RabbitMQ 消息里带当前重试次数和最大重试次数。消费者手动 ACK，解析失败时判断是否达到上限，没达到就重新投递，达到上限就进入死信队列并更新文档状态为失败。死信队列消费者会记录告警日志。管理端还提供按文档重新解析和重建索引的补偿接口。

### Q4：如何处理重复上传？

回答：

> 上传前计算文件 SHA-256 哈希，数据库对 file_hash 建唯一索引。发现已有相同 hash 就直接返回已有文档 ID，不重复上传 OSS 和解析。这样能节省存储和向量化成本。

### Q5：如何保证 ES 和数据库一致性？

回答：

> MySQL 是权威数据源，ES 是检索索引。解析流程先将切片保存到 MySQL，再生成向量并写入 ES，ES 写入成功后才将文档状态更新为解析完成。如果 ES 写入失败，文档状态会失败或保持未完成，可通过重建索引接口从 MySQL 切片重新写 ES。

### Q6：如果 RabbitMQ 消息重复消费怎么办？

回答：

> 消费端先检查文档状态做幂等。如果文档已经处于解析中、解析完成或已发布，就直接 ACK 跳过。切片 ID 也按 docId + 序号生成，重复解析时可以先删除或过期旧切片，再写入新切片。

### Q7：如何处理多版本制度？

回答：

> 同一 displayName 可以有多个 versionNo。发布新版本时，旧版本切片在 ES 中标记为已过期，而不是直接删除，保留审计。检索默认只查已发布、最新版本。管理员可以查询历史版本、发布指定版本、废弃指定版本。

### Q8：RAG 低命中怎么办？

回答：

> 检索结果低于阈值时返回空结果或友好提示，不让模型强答。同时记录低置信检索日志和用户反馈，后续可以用于知识库补全、同义词扩展、切片策略调优和评测集扩充。

### Q9：为什么需要标题路径？

回答：

> 标题路径有两个价值。第一是检索加权，如果问题关键词命中章节标题，这个切片通常更相关；第二是答案溯源，用户看到“某某制度 -> 第三章 -> 第五条 -> 第 6 页”会比只看到一段文本更可信，也方便回到原文核验。

### Q10：为什么写操作要二次确认？

回答：

> 大模型可能误识别、误调用或用户表达不清。银行业务系统里的写操作有真实后果，比如提交请假、审批意见、外包入场申请，所以不能让模型直接落库。二次确认让用户看到结构化摘要和参数，确认后后端再校验一次权限和业务规则，能显著降低误操作风险。

### Q11：安全审计记录哪些内容？

回答：

> 记录操作人、会话 ID、操作类型、工具名称、输入参数、执行结果、拒绝原因、耗时、客户端 IP、traceId、风险等级和时间。敏感字段会脱敏，比如薪资金额、手机号、身份证、银行卡号。审计异步写入，不阻塞主链路。

### Q12：如果用户篡改 pendingOperationId 怎么办？

回答：

> 确认接口会从 Redis 按 userId + pendingId 读取待确认指令，并校验当前用户是否是指令创建者。如果不是创建者，或者指令不存在、已过期、权限已经变化，都会拒绝执行。

### Q13：真实身份系统怎么接？

回答：

> 当前支持网关 Header 身份注入，比如 X-User-Id、X-User-Roles、X-User-Departments、X-Data-Scope。生产环境由统一认证网关或 SSO 校验身份后透传 Header，服务端转换成 EnterpriseUserPrincipal 写入 SecurityContextHolder。业务层统一通过 CurrentUserProvider 获取用户，所以后续替换成 JWT/OAuth2 也不影响业务代码。

### Q14：业务系统怎么接？

回答：

> 业务系统抽象成 Client 接口，包括 LeaveBusinessClient、SalaryBusinessClient、ApprovalBusinessClient、OutsourcingBusinessClient 和 BusinessWriteClient。默认本地用 Mock Client，生产打开配置后用 HTTP Client 调真实业务系统，并透传当前用户身份 Header。这样工具层不用关心底层是真实系统还是 Mock。

### Q15：性能瓶颈在哪里？

回答：

> 主要瓶颈是 Embedding API、Elasticsearch 检索、PDF 解析和大模型生成。在线链路重点优化 RAG 检索，所以做了 BM25/向量并发、查询向量缓存、检索超时降级、舱壁限流和指标监控。离线链路通过 RabbitMQ 异步解析，把耗时任务从上传接口中剥离。

### Q16：如何做高可用？

回答：

> 应用本身无状态或弱状态，状态放在 MySQL、Redis、RabbitMQ 和 ES，可以横向部署多实例。上传解析通过 MQ 削峰，失败进入 DLQ。在线检索有舱壁和超时降级，避免外部依赖异常拖垮服务。生产上 MySQL、Redis、RabbitMQ、ES 应使用集群或托管服务，应用前面挂网关和负载均衡。

### Q17：你如何评估 RAG 效果？

回答：

> 当前已经有检索反馈接口和 eval-samples.jsonl 的评测样例。评估指标可以包括 Recall@K、MRR、引用命中率、低置信率、用户满意度和无答案率。后续会把低命中问题和用户差评自动进入知识补全任务，并用真实制度样本持续调优切片、同义词和重排参数。

### Q18：项目有什么不足？

回答：

> 我会坦诚说当前还有几个方向需要继续生产化。第一，真实银行制度样本不够，PDF 复杂版式、跨页表格和 OCR 还需要调优。第二，业务系统目前通过适配层预留，真实生产要接 SSO、组织架构、权限中心和真实业务 API。第三，RAG 评测集还需要扩充，最好形成离线自动评测流水线。第四，模型输出安全审核和答案强引用校验还可以加强。

## 7. 面试官深挖：模块级追问

### 7.1 RabbitMQ 追问

问：为什么不用线程池异步解析？

答：

> 线程池只能解决单机异步，服务重启后任务可能丢失，也不方便削峰和重试。RabbitMQ 可以持久化消息，天然支持消费者扩展、失败重试和死信队列，更适合文档解析这种耗时、可重试、需要可靠触发的任务。

问：手动 ACK 有什么好处？

答：

> 只有解析成功或确定跳过时才 ACK。解析过程中服务宕机，消息不会丢，可以重新投递。失败达到上限后 nack 不重新入队，进入死信队列，方便后续人工处理或告警。

### 7.2 Elasticsearch 追问

问：dense_vector 为什么是 1536 维？

答：

> 因为使用的是通义 text-embedding-v2，输出向量维度是 1536。ES mapping 中 dense_vector 维度必须和模型输出一致，否则写入会失败。

问：旧版本为什么标记过期而不是删除？

答：

> 银行制度有审计要求，历史版本不能直接物理删除。标记过期可以避免被默认检索，又能保留追溯能力。如果需要节省 ES 存储，可以后续做冷热分层或归档索引。

### 7.3 Redis 追问

问：Redis 存了什么？

答：

> 主要存对话会话状态，包括最近 20 条消息、当前意图、已填槽位、缺失槽位、待确认操作 ID 和会话状态。另外还缓存意图识别结果，TTL 5 分钟；待确认操作也会存 Redis，TTL 5 分钟左右，防止过期确认。

问：为什么会话状态 key 要带 userId？

答：

> 防止串会话。即使两个用户传了相同 conversationId，Redis key 里有 userId 前缀，也不会读到对方状态。

### 7.4 安全追问

问：如果模型生成了越权工具调用怎么办？

答：

> 工具执行前有 BusinessToolPermissionService 做权限校验，校验维度包括角色、本人操作、数据范围和部门。无权限直接返回 permissionDenied，并写审计。写操作即使生成了待确认，确认时还会再次校验当前权限。

问：审计异步写，会不会丢？

答：

> 当前实现是异步写数据库，优点是不阻塞主链路；缺点是极端情况下进程宕机可能丢少量审计。生产上可以改为发送 MQ 或写本地可靠队列，再由审计消费者落库或同步到 SIEM，这是后续增强点。

## 10. 面试时的表达技巧

### 10.1 不要说“我用了大模型”

要说：

> 我把大模型放在意图识别、答案生成和工具调用编排的位置，但业务权限、写操作执行、审计和数据过滤都在后端确定性代码里完成。

这句话很关键，能体现你知道企业 AI 不能完全相信模型。

### 10.2 不要说“RAG 就是向量检索”

要说：

> 我们做的是混合检索，BM25 解决关键词、制度名和条款编号的精确匹配，向量检索解决语义泛化，再用 RRF 融合，并做权限过滤、标题加权、重排和低置信兜底。

### 10.3 不要回避不足

可以说：

> 目前项目已经有生产化骨架，但要真正进银行生产，还需要接真实 SSO/权限中心、真实业务 API、真实制度样本评测集和 SIEM 审计系统。我在设计上已经预留了适配层和配置开关。

这比硬说“已经完全生产可用”更可信。

```text
项目定位：
银行内部 AI 助手 = 制度问答 + 办公办理。
```

## 12. 建议背熟的三句话

1. > 我们把大模型用于理解和生成，把权限、安全、写操作执行都放在后端确定性链路里。
2. > RAG 不是单纯向量检索，而是 BM25 + 向量召回 + RRF 融合 + 权限过滤 + 引用溯源。
3. > 写操作不允许模型直接执行，必须生成待确认指令，用户确认后后端再次校验权限和业务规则。
